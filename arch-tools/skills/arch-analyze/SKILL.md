---
name: arch-analyze
description: Analyze the architecture of one or more projects — generates ubiquitous language docs, technical overview, and optionally an Excalidraw diagram
user-invocable: true
---

Analyze the architecture of the project(s) specified in: $ARGUMENTS

**Parsing arguments**

Parse `$ARGUMENTS` as follows:
- Extract all path arguments (everything that is not a flag). If no paths are given, use the current working directory.
- Check if `--draw` flag is present. If yes, a diagram will also be generated.

Example inputs:
- `/path/to/project` → analyze one project, no diagram
- `/path/to/project --draw` → analyze one project + generate diagram
- `/path/to/projectA /path/to/projectB --draw` → analyze two projects + generate diagram

---

**Step 1 — Explore each project**

For every project path provided:

1. List the top-level directory structure (max 2 levels deep).
2. Detect tech stack by checking for manifests: `package.json`, `go.mod`, `Cargo.toml`, `pom.xml`, `build.gradle`, `pyproject.toml`, `requirements.txt`, `Gemfile`, `composer.json`, `*.csproj`.
3. Read key structural files to understand the domain:
   - Source directories (`src/`, `lib/`, `app/`, `internal/`, `pkg/`, `domain/`, `core/`, `modules/`, etc.)
   - Configuration files (`docker-compose.yml`, `k8s/`, `terraform/`, `.env.example`)
   - Entry points (`main.*`, `index.*`, `app.*`, `server.*`, `cmd/`)
   - README or any docs
4. Identify:
   - **Bounded contexts** or feature modules
   - **Entities and value objects** (models, schemas, types, structs)
   - **Services and use cases** (service files, use-case directories, handlers, controllers)
   - **Infrastructure concerns** (databases, queues, external APIs, caches)
   - **Inter-project dependencies** (if multiple projects given — shared libs, API calls, event contracts)

**Step 4 — Generate Excalidraw diagram (only if `--draw` was given)**

Save the file as `arch-diagram.excalidraw` at the root of the first project path (or current directory).

Generate a valid Excalidraw JSON file that visually represents the **technical flow** from `technical-overview.md`. Follow these rules:

1. Use Excalidraw's JSON format: `{ "type": "excalidraw", "version": 2, "source": "arch-analyze", "elements": [...], "appState": { "gridSize": null, "viewBackgroundColor": "#ffffff" }, "files": {} }`
2. Represent each component as a **rectangle** element with a descriptive label.
3. Represent data flows as **arrow** elements connecting components.
4. Group related components with a **frame** or visually close positioning.
5. Use clear, readable labels. Keep the layout left-to-right for flows, top-to-bottom for layers.
6. Color coding (use `backgroundColor`):
   - Entry points (API, CLI, UI): `#e3f2fd` (light blue)
   - Domain / business logic: `#f3e5f5` (light purple)
   - Infrastructure (DB, queue, cache): `#e8f5e9` (light green)
   - External services: `#fff3e0` (light orange)
   - Inter-project boundaries: `#fce4ec` (light red)

Element template for a rectangle:
```json
{
  "id": "<unique-id>",
  "type": "rectangle",
  "x": <number>,
  "y": <number>,
  "width": 160,
  "height": 60,
  "angle": 0,
  "strokeColor": "#1e1e1e",
  "backgroundColor": "<color>",
  "fillStyle": "solid",
  "strokeWidth": 1,
  "strokeStyle": "solid",
  "roughness": 0,
  "opacity": 100,
  "roundness": { "type": 3 },
  "label": { "text": "<Component Name>", "fontSize": 14, "fontFamily": 1, "textAlign": "center", "verticalAlign": "middle" }
}
```

Element template for an arrow:
```json
{
  "id": "<unique-id>",
  "type": "arrow",
  "x": <start-x>,
  "y": <start-y>,
  "width": <width>,
  "height": <height>,
  "angle": 0,
  "strokeColor": "#1e1e1e",
  "backgroundColor": "transparent",
  "fillStyle": "solid",
  "strokeWidth": 1,
  "strokeStyle": "solid",
  "roughness": 0,
  "opacity": 100,
  "startBinding": { "elementId": "<source-id>", "gap": 8, "focus": 0 },
  "endBinding": { "elementId": "<target-id>", "gap": 8, "focus": 0 },
  "label": { "text": "<optional flow label>", "fontSize": 12 },
  "points": [[0, 0], [<dx>, <dy>]]
}
```

Generate a real, complete JSON — not a placeholder. Base the diagram on actual components and flows discovered during analysis.
